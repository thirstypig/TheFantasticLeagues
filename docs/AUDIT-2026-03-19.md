## 1. Architecture & Structure

### Overall Design

React 18 frontend + Node.js/Express API + PostgreSQL (Supabase). Organized as a monorepo with 17 feature modules that mirror each other on client and server. Each module is self-contained with its own routes, pages, components, API client, and tests.

### Feature Module Completeness

| Category | Modules | Notes |
|----------|---------|-------|
| **Fully Featured (10)** | auth, auction, commissioner, keeper-prep, players, standings, teams, trades, transactions, roster | Server routes + services + tests; client pages + components + API + tests |
| **API-Heavy (2)** | archive, leagues | Full server; client uses API but minimal UI |
| **Minimal/Specialized (5)** | franchises, seasons, periods, waivers, roster | Intentionally lightweight; managed via other modules |

### Module Consistency

- All 17 server features have `routes.ts` + `index.ts` + `__tests__/`
- All routers use named exports (`export const <feature>Router = router;`)
- All API clients use `fetchJsonApi()` from `client/src/api/base.ts`
- No circular dependencies (verified with madge)
- Cross-feature dependencies documented in CLAUDE.md

### Largest Files

No files are dangerously large. The 4 files over 960 lines could benefit from modularization but are not critical.

### Dead Code & Orphaned Files

- **Untracked files in git**: 8 screenshot PNGs, 2 scratch markdown files, and 2 PostHog analytics files (new, integrated into main.tsx). Screenshots are session scratch work, safe to delete.

## 2. Code Quality

### TypeScript

- **Strict mode**: Enabled in both `client/tsconfig.json` and `server/tsconfig.json`
- **`@ts-ignore` / `@ts-expect-error`**: None found in source code
- **`: any` annotations**: Minimal — only in test files (mock signatures) and a few route handlers for flexible object construction. Acceptable.
- **Build errors**: 2 harmless type errors in test files (`__mockFns` export). Tests run fine; these are type-checking artifacts only.

### Naming & Import Conventions

- Server files use `.js` extensions in imports (ESM compat)
- Client files use no extensions in imports
- Router exports: `export const <feature>Router = router;` — consistent across all 17 modules
- Constants: `UPPER_SNAKE_CASE`
- Utility functions: camelCase
- Page components: PascalCase with default exports
- No `console.log` in feature code (only in scripts and `logger.ts`)

### Middleware Ordering

Verified correct in `server/src/index.ts`:
```
attachUser → rate limiting → routes → error handler → SPA catch-all
```

Per-route ordering follows convention:
```
requireAuth → validateBody(schema) → requireSeasonStatus([...]) → requireTeamOwner → asyncHandler(fn)
```

## 3. Security

### Strengths

| Area | Status | Details |
|------|--------|---------|
| **Auth on writes** | All POST/PATCH/DELETE routes use `requireAuth` | No exceptions found |
| **Input validation** | Zod schemas on all mutating endpoints | Proper `validateBody` middleware |
| **SQL injection** | No risk | All queries via Prisma ORM; only 2 `$queryRaw` calls (health check + script) |
| **XSS** | No risk | No `dangerouslySetInnerHTML`, no `eval()`, no `Function()` |
| **Error handling** | Sanitized | Global error handler returns generic messages; details logged server-side |
| **Rate limiting** | Configured | 300/min global, 20/min on auth endpoints |
| **Dev login** | Properly gated | Blocked in production via `NODE_ENV` + requires `DEV_LOGIN_PASSWORD` env var |
| **Secrets in code** | None found | All sensitive values via env vars |

#### Database cascade gaps

8+ foreign key relationships in `prisma/schema.prisma` are missing `onDelete: Cascade`:

| Model | FK | Risk |
|-------|-----|------|
| Team | `leagueId` | Deleting league orphans teams |
| Roster | `teamId`, `playerId` | Orphaned roster entries |
| TeamStatsPeriod | `teamId`, `periodId` | Orphaned stats |
| TeamStatsSeason | `teamId` | Orphaned season stats |
| FinanceLedger | `teamId` | Orphaned financial records |
| AuctionLot | `playerId` | Orphaned lots |
| TradeItem | `playerId` | Orphaned trade items |
| WaiverClaim | `playerId`, `dropPlayerId` | Orphaned claims |
| Period | `leagueId` | Orphaned periods |

**Note**: `AuctionBid` correctly has `onDelete: Cascade`. The others should be reviewed and cascades added where appropriate.

## Key Insight

This codebase has a rare quality for an AI-built project: **compounding institutional knowledge**. The CLAUDE.md + FEEDBACK.md + memory system loop means session 26 builds on everything from sessions 1-25. Most AI projects reset every conversation — this one doesn't, and that's why it's at 46,870+ lines with 670 passing tests across 17 feature modules in 26 sessions.

The architecture is sound, the testing is strong, security is well-handled, and the tooling (slash commands, MCP server, session checklists) supports a sustainable development workflow. The main areas for improvement are operational: CI/CD automation, documentation cleanup, and filling the remaining test gaps.
